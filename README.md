# timebox-app

iPadを中心に、PCブラウザでも利用できる個人用タイムボックスWebアプリです。時間指定／時刻指定の予定、作業／休憩、タイマー、実績を端末内で管理します。サーバー、ログイン、外部API、実行時の外部CDN依存はありません。

公開URL：<https://sazanami3691.github.io/timebox-app/>

## 現在実装済み

### 第1段階：基本版

- 今日・未来日の予定登録、編集、複製、削除取り消し、前日コピー
- 時間指定型／時刻指定型、作業／休憩、時刻重複防止
- タイマー開始、一時停止、再開、完了、スキップ、5分延長
- 終了待ち、超過時間、手動完了、日別履歴
- IndexedDB保存と、終了予定時刻との差による再読み込み・画面復帰
- iPad縦画面優先のレスポンシブUIとPC操作

### 第2段階A：PWA・公開準備

- Web App Manifest、PWAアイコン、GitHub Pagesサブパス対応
- Service Workerによる初回オンライン利用後のオフライン起動
- 新バージョン検出とユーザー操作による更新
- running／paused／expired中の更新保留
- オンライン状態、オフライン準備、アプリ情報画面

### 第2段階B：通知・音・画面点灯維持

- `running`から`expired`へ移った時の大きな終了表示と終了音3回
- 終了音のオン／オフ、音量、ユーザー操作による音のテスト
- Notification APIの利用可否・権限表示、明示操作による権限要求、テスト通知
- 権限がある場合のタイマー終了通知
- 時刻指定予定の「なし／開始時刻／5・10・15分前＋開始時刻」通知
- running中だけ要求するScreen Wake Lockと、状態・対応状況表示

### 第2段階C1：JSONバックアップと全置換復元

- 全期間の予定、確定済み履歴、第2段階B設定をJSONへ書き出し
- Web Share APIのFile共有を優先し、利用できない場合はJSONダウンロード
- 最大10MB、形式・版・全レコード・ID重複・時刻重複を変更前に完全検証
- 件数と作成情報のプレビュー後だけ、IndexedDBの単一トランザクションで全置換
- 実行中タイマーがある場合の復元禁止と、不正ファイル時の既存データ保護

### 第2段階C2：全期間検索と履歴編集

- 全期間の予定・履歴を、作業名またはメモの部分一致で検索
- 前後空白、英字の大文字・小文字、NFKCによる全角・半角英数字の差を吸収
- 予定と履歴を区別して表示し、結果から該当日のカードを開いて強調
- 履歴の作業名、メモ、作業／休憩、完了／スキップ、実績時間を確認後に編集
- 履歴と関連予定を同一IndexedDBトランザクションで同期し、削除後も残存する最新履歴から予定を再調整

### 第2段階C3：長押し並べ替えと階層型設定UI

- 今日・未来日のpending時間指定予定を、専用ハンドルの長押しドラッグで並べ替え
- 400msの長押し成立前に10px移動した場合はキャンセルし、通常の縦スクロールを優先
- ドロップ確定時だけ、対象日のorderをIndexedDBの単一トランザクションで1からの連番へ保存
- タイマー動作中は並べ替え全体を禁止し、保存直前にもcurrentTimerを再確認
- 設定トップの6カテゴリから、1カテゴリずつ全幅の詳細ページを開く階層型UI
- 既存の通知・終了音・予定通知・Wake Lock・PWA更新・バックアップの設定形式と処理を維持

現在のアプリバージョンは`1.5.0`です。

## 長押しドラッグ並べ替え

今日または未来日の「時刻未指定タスク」で、各カード左上の`≡`ハンドルを約400ms長押しし、上下へ動かして離します。カード全体や操作ボタンの長押しでは開始しません。長押し成立前の指移動はスクロールとして扱い、成立後は画面端付近で穏やかに自動スクロールします。

対象はpendingの時間指定（duration）予定だけです。時刻指定（clock）、completed、skipped、過去日の予定、履歴は対象外です。タイマーがrunning、paused、expiredの間はハンドルを無効にします。保存はドロップ時だけ行い、失敗またはキャンセル時はIndexedDBから保存済み順を再取得します。保存された`order`は再読み込み、検索、JSONバックアップと復元でも維持されます。

## 階層型の設定画面

左上メニューから「設定・アプリ情報」を開くと、最初に次の一覧だけを表示します。

1. アプリ更新
2. 通知と終了音
3. 予定通知
4. 画面点灯維持
5. バックアップと復元
6. データ保存とアプリ情報

各行全体をタップすると、そのカテゴリだけを表示する詳細ページへ進みます。上部の「‹ 設定」または、ダイアログが開いていないPC環境ではEscapeキーで一覧へ戻れます。戻ると元の設定行へフォーカスを返します。通知権限要求、音テスト、設定保存、Wake Lock管理、PWA更新、JSON復元ダイアログの既存処理は、詳細ページを閉じていても従来どおり動作します。

## 全期間検索

左上メニューの「検索」を開き、作業名またはメモの一部を入力して「検索」を押します。入力のたびには読み込まず、検索ボタンまたはEnterで全期間の予定と履歴を同じスナップショットから検索します。正規表現や曖昧検索ではありません。結果は予定と履歴を別セクションに表示し、日付の新しい順を基本に、予定は同日内で並び順、履歴は記録時刻の新しい順です。

「該当日を開く」を押すと、予定は日付別予定画面、履歴は日別履歴画面へ移動し、該当カードへスクロールして一時的に強調します。検索結果の表示は最大200件で、超える場合も総件数を表示します。検索語は保存しません。

## 過去履歴の編集と削除

日別履歴の各カードから、作業名、メモ、作業／休憩、完了／スキップ、実績時間（整数分）を編集できます。履歴ID、予定ID、日付、予定時間、実際の開始時刻、記録時刻、記録元、schemaVersionは変更できません。変更内容を確認してから保存し、関連予定があれば作業名・メモ・種類・完了状態・実績時間を原子的に同期します。関連予定が削除済みの孤立履歴も、履歴だけを編集できます。

履歴削除には専用Undoがありません。必ず事前にJSONバックアップを推奨します。関連予定があり、同じplanIdの残存履歴がなければ予定をpendingへ戻し、実績時間と完了時刻を空にします。残存履歴があれば記録時刻が最も新しい履歴（同時刻はIDで安定化）から予定を再構築します。関連予定の日付・状態・必須値が不整合、または同じ予定をcurrentTimerが使用中の場合は、一部変更を残さず中止します。

第2段階C2でも`DB_VERSION`と`SCHEMA_VERSION`は`1`、JSONの`backupVersion`も`1`のままです。新しいストアや必須フィールドを追加せず、C1の書き出し・全置換復元と互換性を維持します。

## 終了音

設定の「通知と終了音」で「終了音を3回再生する」と音量を設定します。初期音量は25%です。先に「音をテスト」を押すと、そのユーザー操作を起点にWeb Audio APIを利用可能にします。外部音声ファイルは使用しません。

タイマーを完了、5分延長、スキップした場合は残りの音を停止します。端末の消音設定、音量、ブラウザの自動再生制限によって鳴らない場合でも、終了待ち画面は必ず表示されます。

## 通知と予定時刻通知

通知権限はページ読み込み時には要求しません。設定画面で次の順に操作します。

1. 「通知を有効にする」を押す。
2. ブラウザまたはOSの確認で許可する。
3. 権限表示が`granted`になったら「通知をテスト」を押す。
4. 時刻指定予定の通知タイミングを選ぶ。

iPadでは、ChromeまたはSafariから追加した「ホーム画面版Webアプリ」を主な利用・確認対象とします。Chrome通常タブ、Safari通常タブ、ホーム画面版ではNotification APIの利用可否が異なる場合があります。Chromeから追加したホーム画面版を利用するためにSafariから再インストールする必要はありません。

予定通知はアプリが動作可能な間だけ確認します。アプリを完全に閉じている間、OSに終了させられた間、長時間バックグラウンドにある間の通知は保証されません。Push API、Push購読、外部通知サーバーは使用していません。予定時刻になってもタイマーは自動開始せず、別タイマーも停止しません。

## 画面点灯維持

設定の「タイマー中は画面を点灯したままにする」をオンにすると、対応環境ではrunning中だけScreen Wake Lockを要求します。一時停止、終了待ち、完了、スキップ、ページ非表示では解除し、表示復帰時にrunningなら再取得を試みます。

Wake LockはOS・省電力設定・ブラウザによって解除または拒否される場合があります。画面が消えてもタイマーの正しさには影響せず、復帰時に現在時刻と終了予定時刻の差から再計算します。

## 保存先と互換性

予定、履歴、現在タイマー、設定はブラウザのIndexedDB `timebox-app` に保存します。DBバージョンは`1`のままで、既存の`plans`、`history`、`currentTimer`、`meta`ストアを維持しています。第2段階B設定と期限付き通知済み記録は既存`meta`ストアへ保存し、古い通知済み記録は件数・期間で整理します。

Service Workerは静的ファイルだけをキャッシュし、IndexedDBやLocalStorageへ触れません。アプリ更新後も既存データと設定を維持します。ただし同じ端末・ブラウザでも、サイトデータを削除した場合やOSがサイトデータを消去した場合は失われる可能性があります。定期的にJSONバックアップを保存してください。

## JSONバックアップと復元

バックアップ対象は、全期間の`plans`、全期間の`history`、終了音オン／オフ、音量、予定通知タイミング、Wake Lock設定です。予定ID、日付、並び順、履歴の孤立したplanIdも保持します。

対象外は、`currentTimer`、ブラウザ／OSの通知権限、`notification-ledger`、Service Worker登録、Cache Storage、更新待ち・Undo・ダイアログなどの一時状態です。実行中でも書き出せますが、そのタイマー状態はJSONへ入りません。

### 書き出し

1. 左上メニューから「設定」を開き、「バックアップと復元」へ移動する。
2. 「バックアップを書き出す」を押す。
3. iPadホーム画面版では共有シートが利用できる場合、「ファイルに保存」など保存先を選ぶ。共有できない環境ではJSONダウンロードへ切り替わる。
4. PCでは共有先またはブラウザのダウンロード先へ保存する。

ファイル名はローカル日時を使った`timebox-app-backup-YYYYMMDD-HHmmss.json`です。内容を外部サーバーや外部APIへ送信しません。

### 全置換で復元

1. 復元前に「バックアップを書き出す」で現在データを保存する。
2. 「復元するJSONファイル」から保存済みファイルを選択する。
3. 全件検証後に表示される作成日時、アプリ版、予定件数、履歴件数を確認する。
4. 警告を確認して「全置換して読み込む」を押す。

復元は追加・マージではなく、現在の予定・履歴・設定をすべて置き換えます。実行中、一時停止中、終了待ちタイマーがある間は実行できません。通知権限、通知済み台帳、実行中タイマーは復元されません。不正JSON、未対応版、不正データ、重複ID・時刻重複、10MB超過は置換前に拒否し、既存データを変更しません。

## iPadのホーム画面版Webアプリ

1. iPadのChromeまたはSafariで公開URLを開く。
2. 共有ボタンまたはブラウザメニューを開く。
3. 利用できる場合は「ホーム画面に追加」を選ぶ。
4. ホーム画面のTimeboxアイコンから起動する。

Chromeから追加したホーム画面版も正式な対象です。初回はオンラインで読み込み、設定画面の「オフライン準備」が準備完了になった後からオフライン起動できます。

## ローカル起動とPC確認

Service WorkerとES Modulesを使うため、ファイルを直接開かずHTTPサーバー経由で起動します。

```powershell
python -m http.server 8000 --bind 127.0.0.1
```

`http://127.0.0.1:8000/`を開き、次を確認します。

1. Consoleエラー、Manifest、Service Worker、`timebox-app-shell-1.5.0`を確認する。
2. 音量を変更して音のテストを行う。
3. 通知権限を許可し、テスト通知を行う。
4. 1分程度のタイマーで終了表示、音3回、終了通知、延長時の音停止を確認する。
5. 時刻指定予定を直近時刻へ登録し、選択した予定通知を確認する。
6. Wake Lock対応ブラウザでrunning／paused／expired時の状態表示を確認する。
7. 予定・履歴・設定を作成してJSONを書き出し、その後の変更が全置換復元で書き出し時点へ戻ることを確認する。
8. タイマー中の復元拒否と、不正JSON選択後も既存データが変わらないことを確認する。
9. 検索でタイトル・メモ・英字大小・全角半角を確認し、予定／履歴結果から該当日へ移動する。
10. JSONバックアップ後、履歴の種類・結果・実績時間を編集し、集計と関連予定が同期することを確認する。
11. 履歴削除のキャンセルと確定を確認し、予定がpendingまたは残存する最新履歴へ戻ることを確認する。
12. 編集済み履歴をJSONへ書き出し、別変更後の全置換復元で書き出し時点へ戻ることを確認する。
13. Offlineへ切り替えて再読み込みし、予定・履歴・検索・編集・設定とバックアップ操作が利用できることを確認する。
14. pendingの時間指定予定を3件以上作り、ハンドルの通常タップでは動かず、長押し後の先頭・末尾・中間移動が再読み込み後も維持されることを確認する。
15. clock、completed、skipped、過去日のカードにハンドルがなく、タイマー中はdurationのハンドルが無効になることを確認する。
16. 設定トップの6行から各詳細へ進み、戻る操作、既存設定の変更、端末回転、キーボードフォーカスを確認する。

自動テストは外部ライブラリなしで実行できます。

```powershell
npm test
```

## iPad実機確認

現在利用中のChromeから追加したホーム画面版を主な確認対象にし、次を確認します。

1. 設定画面が縦・横画面で横にはみ出さずスクロールできる。
2. 音のテスト後、タイマー終了音が3回鳴る。
3. 通知権限を有効化し、テスト通知とタイマー／予定通知が表示される。
4. iPad側の通知設定でTimeboxが許可されている。
5. Wake Lockをオンにしたrunning中の点灯と、一時停止・終了待ち時の解除を確認する。
6. ホーム画面版を閉じた後の通知は保証対象外であることを確認する。
7. JSON書き出し時に共有シートから「ファイルに保存」できること、またはダウンロードへフォールバックすることを確認する。
8. 保存したJSONを選び、プレビュー後の全置換、タイマー中の拒否、不正JSONでの既存データ維持を確認する。
9. 縦・横画面とソフトウェアキーボード表示中に検索し、長いタイトル・メモでも横スクロールしないことを確認する。
10. 検索結果から該当日へ移動し、履歴の編集確認・保存・削除確認を操作する。
11. オフライン状態でも検索と履歴編集が動作し、再起動後も変更が維持されることを確認する。
12. pending時間指定予定を長押しして並べ替え、通常スクロールとの誤操作、自動スクロール、再起動後の順序を縦・横画面で確認する。
13. タイマー中の無効ハンドルと理由表示、clock・完了・スキップ・過去日の対象外表示を確認する。
14. 設定トップの6行をタップして各詳細を開き、戻る、縦スクロール、端末回転、ソフトウェアキーボード表示中の操作を確認する。

必要に応じてChrome通常タブとSafari通常タブも比較しますが、Safariからの再インストールは必須ではありません。

## GitHub Pages設定

公開元を変更する場合はGitHubのSettings → Pages → Deploy from a branchで対象ブランチと`/(root)`を選びます。CodexはPages設定、mainへのマージ、Pull Request作成を行いません。

## 今後追加予定・既知の制限

- 差分／マージ復元、クラウドバックアップ、自動同期、暗号化、自動定期バックアップは未実装です。
- 高度な検索フィルター、日付範囲、曖昧検索、正規表現検索は未実装です。
- 並べ替えはpendingの時間指定予定だけが対象です。時刻指定、完了・スキップ済み、過去日、履歴は並べ替えできません。
- 保存データは使用中のブラウザ・オリジン内だけに存在し、端末間同期はありません。
- 通知、音、Wake Lockはブラウザ、起動方法、iPadOSの状態によって利用できない場合があります。
- アプリを完全に閉じている間の通知は保証されません。
- 過去日の予定本体は閲覧専用です。確定済み履歴は確認付きで編集・削除できます。

第2段階C3でも`DB_VERSION 1`、`SCHEMA_VERSION 1`、`BACKUP_VERSION 1`を維持しています。新しいObject Storeや必須フィールドはなく、C1バックアップおよびC2検索・履歴編集と互換です。

詳細仕様は[`TIMEBOX_SPEC.md`](./TIMEBOX_SPEC.md)、UI基準は[`UI_STYLE_GUIDE.md`](./UI_STYLE_GUIDE.md)、開発ルールは[`AGENTS.md`](./AGENTS.md)を参照してください。
