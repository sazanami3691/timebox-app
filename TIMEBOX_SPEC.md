# timebox-app 仕様書

## 1. 概要

`timebox-app` は、タイムボックス方式で、その日に行う作業と時間を区切って管理する個人用Webアプリである。

ポモドーロ専用には限定せず、作業内容と時間を自由に設定できる。iPadの縦画面を優先し、Safari／Chrome通常タブ、ChromeまたはSafariから追加したホーム画面版Webアプリ、PCブラウザでの利用を想定する。

## 2. 対象外

最小版では次を対象外とする。

- サーバー保存
- ログイン、ユーザーアカウント
- 複数端末の自動同期
- 複数タイマーの同時実行
- 週間カレンダー
- 高度なカテゴリ・タグ管理
- 第1段階でのPWA、通知、終了音、検索、バックアップ

## 3. 技術方針

- HTML / CSS / JavaScript
- 予定、履歴、実行中タイマーはIndexedDBを中心に保存
- 小規模な表示設定はLocalStorage利用可
- 保存データにスキーマバージョンを持たせる
- GitHub Pagesでの公開を予定
- PWA・Service Workerは第2段階

## 4. 画面方針

### 4.1 起動時

- 実行中、一時停止中、終了待ちのタイマーがある場合は、タイマー画面を最初に表示する。
- 実行中のものがなければ、今日の予定画面を表示する。
- 復帰時に終了予定時刻を過ぎていれば、終了待ち画面へ復元する。

### 4.2 ナビゲーション

- 左上のメニューボタンから画面を切り替える。
- 主な項目は「今日」「検索」「履歴」「設定」「実行中タイマーへ戻る」。
- 検索と設定の一部は第2段階で有効化する。

### 4.3 今日・日付別予定

- 日付を前後に切り替えられる。
- 今日だけでなく未来の日付へ予定を登録できる。
- 過去の日付は予定結果・履歴を確認でき、第2段階で編集可能にする。
- 週間画面は作らない。

## 5. タイムボックス登録

### 5.1 登録方式

2種類を使用できる。

1. 時間指定型：30分、60分など
2. 時刻指定型：21:00〜21:30など

### 5.2 表示先

- 時刻指定型は簡易時刻表へ表示する。
- 時間指定型は時刻未指定タスク一覧へ表示する。
- 両方を同じ日の画面で確認できる。

### 5.3 登録項目

- 作業名：必須
- メモ：任意
- 種類：作業／休憩
- 登録方式：時間指定／時刻指定
- 時間、または開始・終了時刻
- 対象日
- 並び順
- 一意ID

### 5.4 時刻重複

- 同じ日の時刻指定予定は重複禁止とする。
- 登録・編集時に既存予定との重複を検証する。
- 重複時は保存せず、重複相手を表示する。

### 5.5 複製

1件単位で複製できる。

引き継ぐもの：

- 作業名
- メモ
- 作業／休憩
- 時間または開始・終了時刻

引き継がないもの：

- 完了・スキップ状態
- 実績時間
- タイマー状態

時刻重複が発生する場合は、修正するまで保存できない。

### 5.6 削除

- 削除後、数秒間「元に戻す」を表示する。
- 元に戻す場合は元の位置へ復元する。
- 実行中、一時停止中、終了待ちのタイムボックスは削除できない。
- 既に記録された履歴は、元の予定を削除しても勝手に消さない。

### 5.7 並べ替え

- 第2段階で長押しドラッグ並べ替えを追加する。
- カード全体ではなく専用ハンドルを使用する。
- iPadのスクロールと競合しないようにする。
- ドロップ完了時にだけ保存する。

第1段階では、登録順または安全な代替操作で順序を管理する。長押しドラッグを先行実装しない。

## 6. タイマー

### 6.1 同時実行

- 同時に動作できるタイマーは1つだけ。
- 一時停止中も、そのタイマーが実行対象を占有する。

### 6.2 基本操作

- 開始
- 一時停止
- 再開
- 完了
- スキップ
- 5分延長

### 6.3 次のタイムボックス

- 現在のタイムボックス終了後、次は自動開始しない。
- ユーザーが「次を開始」を押すまで計測しない。

### 6.4 時刻指定予定の開始

- 予定時刻になっても自動開始しない。
- 第2段階では設定に応じて通知する。
- 別タイマー実行中に次の予定時刻が来ても、現在のタイマーを優先し停止しない。
- 次の予定は開始待ち／予定時刻超過と表示する。

### 6.5 早期開始

時刻指定予定を予定開始時刻より前に開始できる。

例：21:00〜21:30を20:50に開始した場合、20:50から30分計測し、21:20終了予定とする。

### 6.6 遅れて開始

予定開始時刻より遅れているが、予定終了時刻前の場合は開始時に選択する。

- 予定時間をすべて実行
- 予定終了時刻まで実行

予定終了時刻を過ぎている場合は、予定時間をすべて実行する。

### 6.7 開始前確認

- 時間指定型は開始ボタンで即開始。
- 時刻指定型も通常は即開始。
- 遅れて開始し、判断が必要な場合だけ選択ダイアログを出す。

### 6.8 一時停止

- 一時停止中は残り時間を減らさない。
- 一時停止時間は実績作業時間に含めない。
- 最小版の簡易履歴には一時停止時間を別記録しない。
- 一時停止中に閉じても、保存した残り時間を維持する。

### 6.9 終了待ち

- 残り0分で自動完了しない。
- 終了待ち状態へ移行する。
- 「完了」「5分延長」「スキップ」を選ぶまで次を開始できない。
- 第2段階では画面表示、終了音、通知を行う。

### 6.10 超過時間

- 終了待ち中は `+3分` のように超過時間を表示する。
- 完了時に次を選ぶ。
  - 予定時間だけを実績へ記録
  - 超過時間を含めて実績へ記録
- 5分延長時は終了予定時刻を5分後へ更新し、通常の残り時間表示へ戻す。

### 6.11 手動完了

タイマーを使わずに完了できる。

- 実際の作業時間を分単位で入力
- 入力を省略して0分で完了
- キャンセル

入力した時間は合計作業時間へ加算し、達成率では完了として扱う。

### 6.12 実行中の編集

- 作業名とメモだけ編集可能。
- 予定時間、開始時刻、終了時刻は変更不可。
- 時間追加は5分延長を使用する。

### 6.13 アプリを閉じた場合

- 実行中は、アプリを閉じている時間も実時間どおり進める。
- 開始時刻、終了予定時刻、状態を保存する。
- 復帰・再読み込み時に現在時刻との差から再計算する。
- 終了予定を過ぎていれば終了待ちへ復元する。

## 7. 完了・スキップ表示

- 完了・スキップ済み予定は、未完了一覧から外す。
- 同じ日の画面下部に「完了・スキップ済み」欄を設ける。
- 欄は折りたたみ可能にする。
- 完了とスキップを見た目で区別する。
- 実績時間を確認できる。

## 8. 作業と休憩

- 登録時に「作業／休憩」を選べる。
- 休憩も予定表とタイマーで扱う。
- 休憩時間は合計作業時間に含めない。
- 休憩は作業達成率に含めない。
- 日別履歴では、作業とは別に休憩の完了数・スキップ数を表示する。

## 9. 履歴

### 9.1 基本履歴

日別に次を保存する。

- 作業の完了数
- 作業のスキップ数
- 合計作業時間
- 作業達成率
- 休憩の完了数
- 休憩のスキップ数

作業達成率の基本式：

```text
完了した作業数 ÷ その日に実行対象となった作業タイムボックス数
```

延長時間は合計作業時間へ含めるが、達成率には影響させない。

### 9.2 日付またぎ

日付をまたいだタイマーは、全実績を開始日の履歴へ記録する。

### 9.3 過去履歴編集

- 第2段階で過去履歴を編集可能にする。
- 完了／スキップ状態、実績時間、作業名、メモ、作業／休憩、誤記録の削除を対象とする。
- 更新前に確認を表示する。

## 10. 前日の予定コピー

- 前日の予定を選択中の日へコピーできる。
- 作業名、メモ、作業／休憩、時間または時刻、並び順をコピーする。
- 完了・スキップ状態、実績時間はコピーしない。
- コピー先では未開始として作成する。
- 既存予定がある場合は確認する。
- 時刻重複がある場合は、そのまま不正な状態で保存しない。

## 11. 検索（第2段階）

- 作業名とメモの部分一致検索。
- 予定と履歴の両方を対象にする。
- 日付をまたいで検索する。
- 検索結果から該当日を開ける。
- 複雑な状態フィルターは最小版に含めない。

## 12. 通知・終了音（第2段階）

### 12.1 終了時

- 大きな終了表示
- 終了音を3回程度再生し自動停止
- 通知が利用可能な場合は通知
- 操作した場合は残りの音を停止
- 音のオン／オフと音量を設定可能

### 12.2 予定通知

アプリ全体の共通オプションとして次から選ぶ。

- 通知なし
- 開始時刻のみ
- 5分前＋開始時刻
- 10分前＋開始時刻
- 15分前＋開始時刻

予定ごとの個別設定は最小版に含めない。

### 12.3 制限

- サーバーなしの構成では、アプリが完全に閉じている間の確実なバックグラウンド通知は保証しない。
- 通知権限はユーザー操作をきっかけに要求する。
- 音声再生も、事前のユーザー操作が必要になる場合がある。
- 通知や音に依存せず、復帰時に終了済み状態を必ず表示する。

## 13. 画面点灯維持（第2段階）

- 「タイマー中は画面を点灯したままにする」オプションを用意する。
- 初期値はオフ。
- 実行中だけ可能な範囲で有効化する。
- 一時停止、完了、スキップで解除する。
- 非対応環境では通常の自動ロックへ従い、エラーにしない。

## 14. バックアップ（第2段階）

### 14.1 書き出し

JSONファイルへ次を出力する。

- 予定
- 履歴
- 設定
- 前日コピーに必要なデータ
- スキーマ／バックアップ形式バージョン

### 14.2 読み込み

- 現在のデータをすべて置き換える方式だけを採用する。
- 実行中、一時停止中、終了待ちのタイマーがある場合は読み込み不可。
- 実行前に確認し、現在データの書き出しを案内する。
- 形式やバージョンが不正なら、既存データを変更しない。
- 可能なら検証完了後にトランザクションで置き換える。

### 14.3 データ消失への注意

Safariのサイトデータ削除などにより端末内データが消える可能性がある。設定画面などで定期的なバックアップを案内する。

### 14.4 第2段階C1の実装状況

バージョン`1.3.0`でバックアップ形式バージョン`1`のJSON書き出しと全置換復元を実装する。

- `format`は`timebox-app-backup`、`backupVersion`は`1`とし、アプリのschemaVersion・databaseVersion・appVersion・ISO 8601書き出し日時を記録する。
- バックアップ対象は全期間の予定、確定済み履歴、第2段階B設定（終了音、音量、予定通知タイミング、Wake Lock）とする。
- `currentTimer`、通知権限、`notification-ledger`、Service Worker登録、Cache Storage、一時UI状態は対象外とする。
- 読み込みは最大10MBとし、JSON形式、対応版、全予定、全履歴、設定、重複ID、時刻指定予定の重複をIndexedDB変更前に検証する。
- 検証後も直ちに保存せず、作成日時・アプリ版・件数・全置換警告をプレビューし、明示確認後だけ実行する。
- 実行直前に`currentTimer`ストアを同じreadwriteトランザクション内で再確認し、何らかのactiveレコードがあれば復元を拒否する。
- 全置換は既存の`plans`、`history`、`currentTimer`、`meta`ストアを含む単一トランザクションで行い、予定・履歴・タイマーをクリアして検証済みデータと設定を保存し、`notification-ledger`を削除する。無関係なmetaキーは削除しない。
- トランザクションが完了した場合だけ再読み込みし、途中失敗時はabortして既存データを維持する。
- IndexedDBの`DB_VERSION`は`1`のまま維持し、ストアの削除・再作成やデータベース全削除は行わない。

第2段階C2・C3に相当する全期間検索、過去履歴編集、長押しドラッグ並べ替えは未実装のままとする。

## 15. PWA・オフライン・更新（第2段階）

- ホーム画面追加可能なPWA構成を検討する。
- 必要な静的ファイルをオフライン利用できるようにする。
- 新版を検出しても自動再読み込みしない。
- 「新しいバージョンがあります」と表示し、ユーザー操作で更新する。
- 実行中、一時停止中、終了待ちでは更新を保留する。
- 更新後もIndexedDBデータを維持する。
- 設定画面に「更新を確認」を用意する。

### 15.1 第2段階Aの実装状況

バージョン`1.1.0`で次を実装する。

- Web App Manifest、iPad／PWA用アイコン、`.nojekyll`
- GitHub Pagesのプロジェクトサブパスとローカルサーバーで共用できる相対パス
- バージョン付きアプリシェルキャッシュと初回オンライン利用後のオフライン起動
- 新しいService Workerをwaiting状態で保持し、ユーザー操作後だけ適用する更新フロー
- `running`、`paused`、`expired`タイマー中の更新保留
- オンライン／オフライン、Service Worker、オフライン準備、更新状態を表示する設定・アプリ情報画面

Service Workerは予定・履歴・現在タイマーを保存するIndexedDBとLocalStorageへ触れない。今回の実装ではIndexedDBのDBバージョン、ストア、データスキーマを変更しない。

### 15.2 第2段階Bの実装状況

バージョン`1.2.0`で通知・終了音・画面点灯維持を実装する。

- `running`から`expired`へ初めて遷移した終了だけを、画面表示、Web Audio APIによる終了音3回、許可済みNotification APIで案内する。
- 完了、5分延長、スキップでは残りの終了音を停止する。再読み込み時に既に`expired`のタイマーへ復帰した場合は、多重通知防止を優先して再通知しない。
- 通知権限は設定画面の明示操作からだけ要求する。iPadではChromeまたはSafariから追加したホーム画面版Webアプリを主対象とし、通常タブとの差を案内する。
- 時刻指定・未完了予定だけを対象に、なし／開始時刻／5・10・15分前＋開始時刻の共通設定で通知する。確認間隔と猶予時間を限定し、起動前に過ぎた通知を大量送信しない。
- 通知済みキーは予定ID、対象日時、通知種別を含め、既存`meta`ストアへ保存する。14日または500件を上限として古い記録を整理する。
- Screen Wake Lockは設定オンかつ`running`かつ画面表示中だけ取得し、`paused`、`expired`、完了、スキップ、非表示時に解除する。表示復帰時は条件を満たす場合だけ再取得する。

第2段階BでもIndexedDBのDBバージョン、既存ストア、予定・履歴・現在タイマーのスキーマは変更しない。設定と通知済み記録は既存`meta`ストア内でキーを分ける。通知・音・Wake Lockの失敗はタイマー状態遷移やPWA更新処理を停止させない。

## 16. UI要件

- 黒／濃いグレー背景、黒系ウィンドウ、白／薄いグレー枠線。
- 明るい文字、控えめな装飾、高い視認性。
- 重要操作だけアクセントカラー。
- タッチ対象は原則44px以上。
- hoverに依存しない。
- iPad縦画面優先、横画面・PC対応。
- メニューやダイアログを画面外へはみ出させない。
- 必要な場所はスクロール可能。
- 詳細は `UI_STYLE_GUIDE.md` に従う。

## 17. 実装段階

### 第1段階：基本版

- プロジェクト基盤と画面構成
- 時間指定型／時刻指定型の登録・編集・複製
- 作業／休憩
- 簡易時刻表＋時刻未指定タスク一覧
- 時刻重複禁止
- 取り消し可能な削除
- タイマー開始、一時停止、再開、完了、スキップ、5分延長
- 早期開始・遅延開始選択
- 終了待ち・超過時間選択
- 手動完了と実績入力
- 実行中の作業名・メモ編集
- 同時タイマー1つ
- IndexedDB保存
- 復帰・再読み込み時の時刻再計算
- 前日の予定コピー
- 完了・スキップ済み欄
- 基本的な日別履歴
- 縦画面優先レスポンシブUI

### 第2段階A：PWA・公開準備（実装済み）

- PWA、オフライン、ホーム画面追加案内
- 更新検出とタイマー連携した手動更新
- GitHub Pages公開用静的ファイル

### 第2段階B：通知・音・画面点灯維持（実装済み）

- 通知と終了音
- 画面点灯維持

### 第2段階C1：JSONバックアップ（実装済み）

- 全期間の予定・履歴・設定をバックアップ形式バージョン1で書き出す。
- 完全検証とプレビュー後、IndexedDBの単一トランザクションで全置換する。
- `currentTimer`、`notification-ledger`、通知権限はバックアップ対象外とする。

### 第2段階C2・C3以降：未実装

- 長押しドラッグ並べ替え
- 検索
- 過去履歴編集
- GitHub Pages公開設定（ユーザーがGitHub画面で実施）
